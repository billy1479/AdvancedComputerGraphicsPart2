**The Visualisation Pipeline (Lecture 2)** is the backbone of the whole program. The code follows the exact sequence taught: data acquisition (rasterio reads raw LROC TIFFs and extracts geospatial metadata), data filtering (NaN replacement for sentinel values, subsampling for contour performance and 3D surface rendering), data mapping (scalar elevation values mapped to colours, contour lines, displaced 3D surfaces, and Part C's derived quantities like slope and suitability), and rendering (matplotlib produces the final 2D image or 3D perspective view). The `DataStore` and `LunarDataset` classes directly mirror the pipeline's modular decomposition — each stage is a separate responsibility. The three-mode rendering dispatch (`_render_2d`, `_render_3d`, `_render_sidebyside`) shows how the same filtered data can be mapped and rendered through different visual representations while the upstream pipeline stages remain unchanged. Part C extends the filtering stage by computing derived scalar fields (slope, suitability score) from the raw data before mapping them to visual representations — demonstrating that the pipeline is not a single linear path but can branch into multiple derived datasets that are each visualised independently.

**Colour Mapping (Lectures 3–4)** is applied through the colourmap selector, which is shared across both the 2D and 3D views. The lectures warn that rainbow colourmaps have poor perceptual ordering and non-uniform luminance, so the implementation defaults to `terrain` and offers perceptually uniform alternatives like `viridis` and `cividis`. It also includes diverging scales (`coolwarm`, `RdYlBu_r`) which the lectures describe as useful for emphasising deviation from a middle value — relevant here for distinguishing highs and lows relative to a mean elevation. The greyscale option connects to the lecture point that luminance-based maps have natural ordering, even if they sacrifice dynamic range. In the 3D displacement view, the same colourmap is applied per-face to the surface mesh, and when shading is enabled the RGB values are modulated by the hillshade intensity — demonstrating the lecture's point that per-pixel colour mapping (applying the colourmap after interpolation, not before) keeps all colours within the colourmap's range and avoids the interpolation artifacts discussed in Lecture 4's colourmap implementation section. Part C introduces additional purpose-built colourmaps: the slope overlay uses a custom green→yellow→red sequential map designed so that the colour directly encodes safety (green = flat/safe, red = dangerously steep), while the suitability heatmap uses `RdYlGn` as a diverging scale where the midpoint (yellow) represents marginal sites and the extremes (red/green) represent poor/excellent candidates. These choices follow the lecture's colourmap design principle of matching the colour semantics to the data semantics.

**Height Plots and Displacement Maps (Lecture 4)** are implemented directly in `_render_3d()`. The lectures define the displacement formula S_displ(x) = x + n(x)·f(x), where a surface S is displaced along its normal by the scalar value. For terrain, S is the xy-plane and displacement is along z, which is exactly what `plot_surface(X, Y, Z * vexag)` computes — each grid point is lifted by its elevation value scaled by the vertical exaggeration factor. The lectures specifically note that shading in height plots acts as an additional perceptual cue, emphasising fine-grained (small-scale) data variations that would be invisible in a flat colour map. The `_compute_hillshade()` function implements this by computing the surface gradient (∂Z/∂x, ∂Z/∂y), deriving slope and aspect, and calculating Lambertian illumination from a virtual light source at a fixed azimuth and altitude. This shading is then multiplied into the colourmap's RGB channels, so the viewer perceives both elevation-as-colour and local terrain roughness-as-shading simultaneously. The user controls viewing azimuth (-180° to 180°), elevation angle (5° to 90°), and vertical exaggeration (0.5× to 15×), allowing them to rotate around the terrain and adjust how prominently vertical features appear — all parameters the brief explicitly requires.

**Contouring / Isolines (Lecture 4)** is implemented in both 2D and 3D contexts. The lectures define a contour as I(f₀) = {x ∈ D | f(x) = f₀} — the set of all points having a particular scalar value. Key properties from the lectures: contours are always closed curves (unless they exit the domain) and they never self-intersect, which is why they produce clean, nested elevation lines on the lunar terrain. The marching squares algorithm — which encodes the inside/outside topological state of each cell vertex in a 4-bit code and uses a 16-case lookup table — is the standard approach for computing contours on 2D quad grids; matplotlib's `contour()` function implements this internally. The `_overlay_contours_2d()` method generates isolines at user-specified intervals on the flat map, while `_draw_contours_3d()` projects the same contour lines onto the displaced 3D surface, letting users see how abstract isolines correspond to physical ridges and valleys. GUI controls let users adjust the contour interval (which isovalues are generated), line width, line colour, and whether numerical labels are shown — exactly the parameter control the brief requires. Part C extends contouring further: the slope overlay draws a dashed red contour at the user-defined danger threshold, and the PSR overlay draws a solid boundary contour around permanently shadowed regions — both using the same underlying isoline computation on derived scalar fields rather than raw elevation.

**Colour Banding and Contour Duality (Lecture 4)** is exploited deliberately in the 2D view. The lectures show that colour banding (discretising the colour scale into a finite number of bands) is mathematically related to contouring — the transition between two consecutive colour bands is essentially a contour. The implementation combines smooth colour variation (continuous colourmap) with overlaid contour lines: the colour gives a rapid qualitative overview of the elevation field while the contours let users trace specific elevation levels precisely. Users can observe this duality directly by adjusting the contour interval slider: a very small interval produces dense lines that approximate the visual effect of colour banding, while a large interval gives sparse, widely-spaced isolines that highlight only the major elevation transitions.

**Side-by-Side Comparison (Lecture 4, applied)** is the core Part B deliverable, implemented in `_render_sidebyside()`. The lectures present 2D colour-mapped contour representations and 3D displacement plots as complementary techniques with distinct trade-offs. The side-by-side view renders both simultaneously — contour map on the left, 3D displacement map on the right — with a shared colour bar, so the user can directly evaluate these trade-offs. The 2D contour view preserves spatial accuracy and supports precise quantitative reading (exact elevation at any isoline) but is abstract and requires learned interpretation. The 3D displacement view reveals terrain shape intuitively through shading and perspective, making it easy to spot craters, ridges, and flat areas at a glance, but introduces occlusion (back-facing slopes are hidden), perspective distortion (foreshortening varies across the view), and loss of precise spatial measurement. The three-mode radio button toggle (2D only, 3D only, side-by-side) lets users switch between these representations as their analysis task demands — overview exploration in 2D, intuitive terrain assessment in 3D, and direct comparison in the combined view. Part C overlays (slope, PSR, suitability) appear on whichever 2D panel is active, so users can evaluate these derived quantities in the context of both visualisation techniques.

**Derived Scalar Quantities from the Gradient (Lectures 4–5)** are the foundation of Part C's slope analysis. The lectures cover computing derived quantities from scalar fields, particularly the gradient ∇f which gives the direction of steepest ascent. `_compute_slope_grid()` applies Sobel operators to compute ∂f/∂x and ∂f/∂y from the elevation grid, then calculates slope magnitude as arctan(|∇f|) in degrees. This is a direct application of the lecture concept: transforming a raw scalar field into a derived scalar field that encodes a different physical quantity (terrain steepness rather than absolute height). The slope is critical for landing site safety — slopes above ~15° are considered dangerous for lunar landers — so visualising this derived quantity directly supports the mission planning use case. The Sobel filter's 8-pixel neighbourhood provides noise-robust gradient estimation compared to simple finite differences, and the pixel spacing from the rasterio metadata is used to ensure the gradient has correct physical units.

**Information Layering and Multi-variate Visualisation (Lectures 3–4)** is applied through multiple overlapping layers. The lectures discuss the challenge of displaying multiple scalar fields on the same domain, and emphasise that effective visualisation supports comparisons and tells stories about the data. The implementation uses several complementary approaches. Transparency blending layers illumination over elevation — the base colourmap shows terrain while the semi-transparent overlay shows solar conditions, letting users visually correlate both. PSR regions are drawn as semi-transparent blue polygons with boundary contours, creating a categorical overlay (shadowed vs. illuminated) on top of the continuous elevation field. The slope overlay uses a separate colourmap at adjustable opacity so the user can tune how prominently safety information appears relative to the terrain. Each overlay adds a new dimension of information without replacing the underlying data — the lectures describe this as the key challenge of multi-variate visualisation, and the adjustable opacity sliders let users manage the visual complexity by fading layers in and out as needed. The click-to-sample readout extends the inverse mapping concept to report all active variables at once (elevation, illumination, slope, suitability), giving the user a complete multi-variate picture at any point.

**Inverse Mapping (Lecture 2)** is what the click-to-sample feature implements. The lectures describe visualisation as a function v: D^m → R^5 (data to image), and analysis as the inverse v⁻¹: D^5 → R^m (image back to data). When the user clicks on a coloured pixel and the code inverts the affine transform to retrieve the underlying elevation and illumination values, that's exactly this inverse mapping in action. This works in both the standalone 2D view and the left panel of the side-by-side comparison, giving users quantitative readout alongside both visual representations. Part C extends the readout to include slope angle (with a "STEEP!" warning when the danger threshold is exceeded) and the composite suitability score, so the inverse mapping now recovers not just the raw data but also the derived quantities — demonstrating that inverse mapping applies equally to any scalar field, whether original or computed.

**Multi-scale / Overview-to-Detail (Lectures 1–2)** is addressed by the three spatial scale levels and the zoom mechanism. The lectures emphasise that effective visualisation presents information at several levels of detail, from overviews to fine structure. The radio buttons switch between regional, intermediate, and detailed datasets, while the right-drag zoom lets users drill further into sub-regions within any scale — implementing semantic zoom where contour density and detail adapt to the viewed extent. Scale changes propagate to both the 2D and 3D views, so users can examine the same region at the same scale using different representation techniques. Part C's derived layers (slope, PSR, suitability) are recomputed for each scale level, so the analysis adapts to the resolution of the data — at the regional scale the slope map reveals large geological features, while at the detailed scale it highlights individual crater rims and boulder fields.

**Filtering and Slicing (Lectures 2–3)** is applied in Part C's elevation profile cross-section. The lectures describe filtering as extracting subsets of data to reveal structure that isn't visible in the full dataset. When the user clicks two points on the 2D map, the code samples elevation (and illumination) along the connecting line to produce a 1D cross-section — a slice through the 2D scalar field. This is displayed in a popup window as an elevation profile plot, with the terrain filled below the curve and dangerous slope segments highlighted in red. The profile reveals vertical structure (crater depth, ridge height, slope steepness) that is difficult to perceive in the overhead 2D view and may be occluded in the 3D view. If illumination data is available, a second panel shows the illumination profile along the same transect, allowing users to correlate terrain shape with solar conditions at a glance. This directly connects to the lecture concept that different visual representations of the same data reveal different aspects of the underlying phenomenon.

**Effective Colourmap Design Goals (Lecture 3)** shaped specific choices across all three parts. The lectures list five goals: telling absolute values, value ordering, value differences, identifying selected values, and showing rate of change. The terrain colourmap handles ordering well through luminance progression. Contour lines address the "selected values" goal by marking exact elevation levels. The illumination overlay with a sequential `hot` colourmap prioritises ordering (dark = shadowed, bright = illuminated) — matching the lecture advice to use luminance for clear ordering in compact domains. In the 3D view, the hillshade modulation adds a sixth visual channel (brightness variation from shading) that encodes the rate of elevation change (gradient magnitude and direction) — directly addressing the "value change" design goal that the lectures identify as one of the hardest to achieve with colour alone. Part C's slope colourmap is designed specifically for the "specific value ranges" goal: the green→yellow→red progression creates an immediate visual distinction between safe (<threshold) and dangerous (>threshold) terrain, and the dashed contour at the threshold itself addresses the "specific values" goal. The suitability score's `RdYlGn` diverging map addresses "value ordering" and "value differences" simultaneously — users can both rank candidate sites (greener = better) and estimate how much better one site is than another by the colour distance.

**Composite Scoring as Multi-variate Data Mapping (Lectures 3–4)** is Part C's most advanced application of lecture concepts. The suitability score combines three normalised scalar fields — flatness (derived from slope gradient), illumination (from the LROC data), and PSR proximity (from a distance transform of the shadow threshold) — into a single composite score using user-adjustable weights. This directly demonstrates the lecture concept of mapping multi-dimensional data to a visual representation: three separate scalar fields are reduced to one through weighted combination, which can then be visualised as a standard colourmap overlay. The distance transform used for PSR proximity is itself a derived scalar field — computing the Euclidean distance from each non-shadowed pixel to the nearest shadowed region — which creates a smooth gradient from the PSR boundary outward. The Gaussian decay applied to this distance encodes the scientific trade-off: being near a PSR is valuable (potential water ice access) but being inside one is dangerous (no solar power). The weight sliders let users explore different prioritisations interactively, making the visualisation a genuine decision-support tool rather than a static display.
